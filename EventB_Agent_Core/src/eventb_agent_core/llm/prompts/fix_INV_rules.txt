5. If the goal and its most similar hypothesis in the node partially match one of the `\\forall` or `\\exists` axioms, use `selectHypothesisFromContext` to add the hypothesis to the proof tree.
6. For an invariant-preserving proof obligation (***INV***), if proof cannot proceed, use `addHypothesesToContext` OR `addHypothesesToGuard`. 
6a. Event-B is based on proof by induction. For an obvious fact, e.g. `x >= 0` |- `x + 1 >= 0` that is an ***inductive step***, and not basic knowledge of Event-B auto prover, use `addHypothesesToContext` to add an axiom, e.g. `\\forall x \\cdot x \\in NAT \\land x >= 0 \\implies x + 1 >= 0`. Add an hypothesis specific to the proof. After this, specify instantiations specific to the current proof.
6b. The added hypothesis ***must be relevant to the goal*** of the selected node. e.g. If there are many selected hypothesis, and the goal is `x+1 \\in NAT`. Then you should focus on the selected hypothesis such as `x \\in NAT`, which is relevant to the goal by an inductive step.
6c. For ***INV***, if the proof involves parameters of an event, use `addHypothesesToGuard` to add the inductive step as an additional guard. e.g. if `p` is a parameter of the event, and `f \\in TYPE` |- `f \\oplus {p \\mapsto n} \\in TYPE`, then use `addHypothesesToGuard` to add a guard, e.g. `\\forall f \\cdot f \\in TYPE \\land p \\in dom(f) \\implies f \\oplus {p \\mapsto n} \\in TYPE`.
6d. If the hypothesis involves multiple variables to be instantiated, used the format `\\forall x,y \\cdot ...` and `\\exists x,y \\cdot ...`, instead of `\\forall x \\cdot ... \\implies \\forall y ...`.
6e. After adding hypothesis that is `\\forall` or `\\exists`, you also need to specify instantiations. e.g. `\\forall x,y,z ...`, the instantiations array should contain 3 items for `x`, `y`, `z` respectively, and the order must match.
7. If add hypothesis (`ah`) appeared in the proof tree, but its children nodes are not discharged, use `instantiation` to discharge the children nodes.
